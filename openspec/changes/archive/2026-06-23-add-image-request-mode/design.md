## Context

生图能力已有多个入口：配置与用户偏好中的模型设置、生图工作台、无限画布生成配置节点、普通文本/图片节点的生图参数弹窗。现有请求集中在 `web/src/services/api/image.ts`，配置集中在 `web/src/stores/use-config-store.ts`，画布节点私有配置存放在 metadata 中。

本次变更横跨全局 store、API 请求层、工作台页面和画布组件，因此需要明确状态归属和覆盖规则。

## Goals / Non-Goals

**Goals:**

- 增加全局默认生图请求方式，默认流式请求。
- 在所有生图入口提供“沿用总配置 / 同步请求 / 流式请求”选择。
- 保留同步请求路径，流式请求仅在用户配置选择时启用。
- 保存实际请求方式，保证历史记录和画布重试行为一致。

**Non-Goals:**

- 不改变视频、音频和文本生成请求方式。
- 不为 Gemini 图片生成新增流式能力。
- 不实现流式过程中的图片预览，只解析最终图片结果。
- 不调整渠道、模型、API Key 的安全边界；API Key 仍保存在浏览器本地并由前端直连 OpenAI 兼容接口。

## Decisions

### Decision: 全局配置用 `imageRequestMode`，入口覆盖用 `"global" | "sync" | "stream"`

全局配置只保存实际可执行值 `"sync"` 或 `"stream"`；工作台和画布 UI 使用 `"global"` 表示沿用总配置。这样全局配置始终简单可执行，入口层负责解析覆盖关系。

替代方案是所有入口都直接保存实际值。该方案会让“沿用总配置”变成某一刻的固定值，用户后续修改总配置时旧节点不会跟随变化，不符合截图需求。

### Decision: 画布节点 metadata 只在覆盖时保存请求方式

画布生成配置节点和普通节点参数弹窗选择“沿用总配置”时清除节点覆盖值；选择同步/流式时写入 `metadata.imageRequestMode`。生成结果 metadata 保存实际使用值，用于重试。

替代方案是在创建节点时写入默认请求方式。该方案会使新节点过早固定配置，削弱全局默认配置的意义。

### Decision: 请求层保留 axios 同步路径，新增 fetch SSE 路径

同步请求继续使用现有 axios 逻辑，减少行为变化；流式请求使用 `fetch` 读取 `text/event-stream`，并复用已有错误解析风格。SSE 解析兼容最终 `data` 数组、`b64_json`、`url` 以及常见 inline image 字段。

替代方案是把同步和流式统一改成 fetch。该方案会扩大风险面，可能影响现有同步渠道兼容性。

## Risks / Trade-offs

- [Risk] 不同 OpenAI 兼容服务的图片流式事件字段不完全一致。→ 解析器兼容最终 payload 与常见图片字段；无法解析时仍抛出明确错误。
- [Risk] 流式请求目前只在完成后展示最终图片，没有中途预览。→ 保持改动最小，避免牵动结果面板和画布节点的加载状态模型。
- [Risk] 项目已有 TypeScript 严格检查错误会影响单独 `tsc --noEmit`。→ 本次验证以 `npm run build` 为准，同时记录既有 `tsc` 错误。

## Migration Plan

- 旧浏览器本地配置缺少 `imageRequestMode` 时，store merge 阶段归一化为 `"stream"`。
- 旧生成记录和旧画布节点缺少请求方式时，按全局配置或默认流式请求解析。
- 不需要服务端迁移或数据库迁移。

## Open Questions

- 后续是否需要在流式请求过程中显示 partial image 预览。
- 后续是否需要按渠道配置默认请求方式，而不是仅提供全局默认。
